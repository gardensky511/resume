<ins>大規模フロントエンドにおける状態管理のリファクタリング</ins>
- 課題・背景
  - Redux の store がページ単位で生成され定義が不統一なため、状態不整合による障害が頻発
  - 共通処理における TypeScript の型定義が不正確（`any` の多用）で、200箇所以上の潜在的な実行時エラーが存在
  - メンテナンス停止ライブラリ（`next-redux-wrapper`）への依存により、Next.js のバージョンアップが不可能な状態
- 行動・成果
  - 安全性を重視した段階的リファクタリング計画を立案し、影響範囲 41 ページに及ぶ大規模改修を障害なくリリース
  - 従来の手動テストではテストに数ヶ月を要する状況だったため、単体テストで安全性を担保できる体制を提案・構築し、テスト工数を大幅削減
  - 全ページの store 定義を統一し、store 不整合起因の障害を根本的に解消
  - 200箇所以上の潜在的実行時エラーを解消し、アプリケーションの堅牢性を大幅に向上
  - `next-redux-wrapper` の機能を自作処理で代替し、Next.js のバージョンアップが可能な状態を実現

<ins>TypeScript 型安全化（`any`撲滅）による品質改善</ins>
- 課題・背景
  - `noImplicitAny: false` により暗黙的 `any` が許容、明示的 `any` も一部許容されランタイムエラーが発生
  - `any` 全面禁止を試行すると約 2,000 件の型エラーが発生し、短期一括対応は期間・影響ともに過大
  - 大規模なコード変更はリリースロールバックのリスクが高く、対応が遅れるほど新規の `any` が増加する懸念
- 行動・成果
  - 「挙動非変更（型とコメントのみ修正）」を原則とする段階的リファクタリング方針を策定
    - (1) 可能な箇所は正確な型を付与
    - (2) コード修正が必要な箇所は一旦 `@ts-ignore` で退避し、TODO コメントに解消方針を明記（根本対応を後続タスク化）
  - 既存の型エラーを上記方針で段階的に解消し、`noImplicitAny: true` の有効化、ESLintルールの変更を迅速にリリースし、暗黙的・明示的 `any` を全面禁止
  - これにより、挙動を変えず短サイクルで設定をリリースし、将来の型整備を継続可能にする安全で堅牢な開発基盤を確立
