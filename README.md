<ins>大規模フロントエンドにおける状態管理のリファクタリング</ins>
- 課題
  - store が不統一な定義でページごとに生成されるかつ、全コンポーネントから store に直接アクセスされている構造が原因で障害発生。
  - SP/PC（2つのアプリ）間で状態が共通化された結果、型定義が放棄され `any` 型が多発し、潜在的エラーが200件以上存在。
  - メンテナンス停止ライブラリ（`next-redux-wrapper`）への依存により、Next.js を2年分アップデート不可。
- 工夫
  - 41ページに至るほど影響範囲が大きいため、理想への一括移行はリスクが高いと判断。安全性を最優先し、優先順位高い課題を以下の3ステップの段階的リファクタリングを計画・主導。
    - (1) store 定義の統一
      - 障害の根本原因である store 定義の不統一を解消。全 store 定義をマージした共通 store 定義を作成し、各ページはそれを参照して store を生成させることで1アプリ1store"定義"実現。
    - (2) `any` 型問題の解消
      - 共通化されたセレクター側で `<S extends UiState>` のようにジェネリックで安全のための最低限の型を強制し、アプリケーション側では型付けされたセレクター（`useAppSelector`）を使うことで型エラーを探知できる状態を実現。
    - (3) レガシーライブラリの排除
      - 最速での排除を優先し、ライブラリの機能（サーバー側 store をクライアント側の store の初期値にする）を自作処理に置き換えることでレガシーライブラリをは排除。
  - 最大の障壁は数ヶ月を要する手動テストだったため、安全を担保できる単体テストを新たに作成することで手動テストを最小限にする案を考案し、チームの合意形成に成功。
- 成果
  - 上記戦略により、影響範囲41ページの大規模改修を3段階に分けて障害なく完遂。
  - テスト方針の変更により、テスト工数を数ヶ月から数週へと大幅に削減。
  - store 不統一に起因する障害を根本的に解消し、関連障害0件を実現。
  - 200箇所以上の潜在的実行時エラーを解消し、アプリケーションの堅牢性を大幅に向上。
  - `next-redux-wrapper` の機能を自作処理で代替し、Next.js のバージョンアップが可能な状態を実現。

<ins>TypeScript 型安全化（`any`撲滅）による品質改善</ins>
- 課題
  - TS設定（`noImplicitAny: false`）とリント回避（`@ts-ignore`）の多用により、暗黙的・明示的な `any` が約2,000箇所に多発。
  - 大規模・大人数での開発体制のため、設定を変更しなければ日々 `any` が増加し続ける懸念があった。
- 工夫
  - 2,000件の一括修正は工数・人員的に非現実的であり、もし修正が原因でリリースが差し戻されれば、その分 `any` が増え続ける期間が延びると判断。
  - まずは「設定変更による `any` 増加防止」を最優先のゴールに設定し、デグレードリスクを完全にゼロにするためランタイムの挙動は一切変えず型とコメントのみを修正するという原則設定。
  - 既存の2,000件の型エラーに対しては以下の優先順位で対応
    - (1) 可能な箇所は正確な型を付与
    - (2) コード修正が必要な箇所は一旦 `@ts-ignore` で退避し、TODO コメントに解消方針を明記（根本対応を後続タスク化）
  - リント回避（`@ts-ignore`）による `any` 使用を防止するために `any` を禁止するルールはリント回避できない設定も追加
- 成果
  - 約2,000件の既存エラーを段階的に解消し、`any` を全面的に禁止する設定を迅速にリリースすることに成功。
  - 新たな `any` が今後増えることを恒久的に防止する仕組みを確立し、アプリケーションの安全性を大幅に向上。

<ins>リリース作業とリリースノート作成自動化</ins>
- 課題
  - UIライブラリのリリース、ノート作成、Storybook 反映が全て手作業で、1回に40分以上を要し、リリース頻度が低下していた。
- 工夫

  - 最大の障壁は、GitHub Actions（Linux環境）からデプロイ先（Windows共有フォルダ）へのStorybook更新だった。このOS間の差異は、`cifs-utils` を用いて GitHub Actions 側に Windows 共有フォルダをマウントすることで解決し、プロセスの完全自動化に成功。
- 行動・成果
  - リリース、ノート作成、Storybook 更新というリリースプロセス全体を自動化し、リリース作業時間を90%以上短縮。
  - 従来は5〜6PRをまとめて1回リリースしていたが、自動化により1PR単位でのリリースが可能になり、リリース頻度が5〜6倍に増加。

買取店舗で使用する業務システムの Web フロントエンド開発を担当。

<ins>開発環境・チーム体制</ins>
- 開発環境：React、Next.js（Pages Router）、TypeScript、GraphQL、React Hook Form、Zod、Material UI
- チーム体制：PM 1 名、開発者（フロントエンド 3 名、バックエンド 2〜3 名、QA 1 名）

<ins>E2Eテストケース実行時間短縮</ins>
- 課題
  - MagicPod（E2Eテストツール）でチェックボックスやラジオボタンがあるページのテスト実行時間が長い（１つをチェック済みにするのに4秒以上かかる）。
  - MagicPod側への問い合わせたら「カスタムスタイリングのために `input` タグに `opacity:0`（透明化）を設定していたことが原因で、要素の検出に時間がかかっている」という回答があった。
- 工夫
  - `input` タグが透明になっていることが原因なら透明にせず `input` タグのデフォルトデザインを全部消すアプローチを考案、原因となっていた `opacity: 0` を `all: unset` に差し替えた。
- 成果
  - 仮説は正しく MagicPod が即座に要素を検出できるようになり、対象ページのE2Eテスト所要時間を約67%以上大幅に短縮。
